# Generating & Executing Kilo Plans

## Purpose

Explain how to produce a structured execution plan with the **plan** sub‑agent and how to run that plan with an orchestrator. The process is idempotent, supports explicit model selection, and includes built‑in error handling.

## 1. Generate a plan

1. **Define the goal** – a short, high‑level statement (e.g. "Add a dark‑mode toggle to the web UI").
2. **Invoke the planner** using `kilo run` (or the CLI shortcut `kilo task` if configured):

   ```bash
   kilo run \
     --description "Generate plan for <goal>" \
     --prompt "<goal>" \
     --agent plan
   ```

   Replace `<goal>` with your actual objective.
3. The planner returns JSON following the schema in `agent/plan.md`. Key fields:
   * `goal` – original request
   * `idempotent` – true if the whole plan may be safely re‑run
   * `steps` – ordered array where each step contains:
     * `id` – unique identifier (e.g. `step-1`)
     * `description`
     * `model` – any builtin shortcut (`reason`, `ask`, `chat`, `code`, `coder-qwen`) **or** a full provider/model identifier such as `azure/gpt-4` or `openai/gpt-4o`
     * `input` / `output`
     * `gate`, `parallel`, `retry`, `max_retries`, `on_error`, `self_check`


## 3. Execute the plan

Use the orchestrator (the primary `orchestrate` agent) to consume the JSON:

```bash
kilo run \
  --description "Execute generated plan" \
  --prompt "$(cat plan.json)" \
  --agent orchestrate
```

The orchestrator:

* Reads each step in order.
* Calls the specified model/sub‑agent.
* Runs `self_check` after completion.
* Honors `gate` (manual approval), `parallel` (concurrent execution), and retry logic.
* Stops or continues based on `on_error`.

Because the plan is marked `idempotent`, you may safely re‑run the whole plan—or only the failing steps—without side‑effects, provided the underlying actions themselves are idempotent (e.g., use upserts, check‑before‑write).

## 4. Model‑selection policy

* Prefer builtin shortcuts for simplicity.
* For performance‑critical or accuracy‑critical tasks, specify an explicit provider/model (`azure/gpt-4`, `openai/gpt-4o`).
* The planner defaults to the free/open‑weight model unless a higher‑capability model is explicitly required.
